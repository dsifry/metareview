Evidence before done

Review gates that make agentic work safer to ship.

metareview helps humans and coding agents catch missing acceptance evidence, duplicated codepaths, intent drift, and PR-readiness gaps before work is called done.

0.6.0 adds evidence receipts, context profiles, and review manifests.

The 0.4.0 to 0.6.0 work focused on making review gates more actionable for real agents working in large or messy repos.

Receipts: validation commands and GitHub checks can be captured as structured evidence with exit codes, timestamps, summaries, and output hashes.

Profiles and shards: task-done, epic-ready, and PR-ready reviews now record diff size, generated artifact exclusions, omitted or truncated context, and deterministic shard plans when the review surface is too large or risky.

Manifests: Task-done and PR-ready Review Manifest sections account for source paths, generated dispositions, shard coverage, source-manifest hashes, and aggregate blockers.

The problem is not that agents cannot write code.

The problem is that agents are too willing to declare victory.

A coding agent can finish a task, summarize what it thinks it did, and sound confident while missing the acceptance contract, duplicating an existing service path, skipping a test, or drifting away from the original intent.

Humans compensate by doing the orchestration themselves: rereading the spec, checking the diff, asking for tests, reviewing the plan, comparing back to the parent issue, and remembering the repo-specific gotchas the agent forgot.

metareview turns that work into named review gates with durable artifacts, blocker policy, repository knowledge, and post-merge learning.

A review is not a vibe check. It is a state transition.

The review pipeline

Use the smallest gate that matches the decision in front of you. Review early, review at boundaries, and keep blockers as work until a fresh pass clears them.

1

Artifact Review

Review specs, implementation plans, architecture notes, designs, decomposition docs, and other written intent before agents start coding.

2

Task-Done Review

Review a local task-sized change before an agent says it is done. Check acceptance, tests, diff scope, service boundaries, and evidence.

3

Epic-Ready Review

When child tasks pass, review whether the parent is actually ready. Look for integration gaps, drift, missing rollback paths, and partial completion hidden across subtasks.

4

PR-Ready Review

Before push, PR creation, or merge readiness, inspect the whole branch as a product: intent, diff, tests, docs, open findings, and GitHub context.

5

Post-Merge Learning

After merge, extract useful lessons, calibrate reviewers, update local knowledge, and prune stale or obvious notes before they become noise.

Fractal review, not one big pass at the end

Large work decomposes. The reviews should decompose too.

Original intent → parent plan → child plans → task changes → parent intent check
Plan

Review the parent

Is the plan feasible, complete, testable, and aligned with the user request?

Decompose

Review the children

Do task boundaries make sense? Are dependencies explicit? Does every child have acceptance criteria?

Block

Fix and re-run

Blocking findings are not commentary. They are current work until a fresh review clears them.

Reconcile

Return to intent

After iterations, compare the accepted work back to the original goal so local fixes did not change the mission.

It learns the repo instead of pretending every repo is the same.

metareview is local-first. It looks for the knowledge your team already wrote down and adds to it when reviews teach something durable.

Knowledge sources: service inventories, Beads knowledge, prior review artifacts, git history, GitHub PR context, session history, and local docs.

Knowledge outputs: Markdown review files under docs/metareview/, local JSONL run/finding state under .metareview/, and curated learning for future reviews.

Pruning rule: do not keep facts just because they are specific. Stale, self-evident, and overly narrow notes should be discarded.

Use cases

metareview is useful whenever the next step should depend on evidence instead of confidence.

Before coding

Spec and plan review

Challenge requirements, architecture, feasibility, failure paths, scope, and acceptance before implementation starts.

During execution

Task-done review

Review small local code chunks while the agent still has context and before the work is allowed to advance.

Across epics

Decomposition review

Review parent/child task boundaries, dependency order, acceptance coverage, and integration risk.

Before push

PR readiness

Check the branch against original intent, tests, docs, review evidence, and open blockers before asking others to review.

After merge

Learning capture

Turn review feedback and repeated corrections into repo knowledge that future agents can retrieve.

Always

Service boundary protection

Use service inventories and local knowledge to catch duplicated codepaths and parallel services before they harden.

Review patterns

The useful patterns are simple, but they have to be enforced.

  • Adversarial multi-reviewer passes with named perspectives.
  • Hard blockers for critical findings, spec-contract failures, and acceptance gaps.
  • Fresh re-review with --previous-run instead of self-certification.
  • Evidence-backed decisions: diff, tests, acceptance notes, and GitHub context.
  • Fractal review loops for decomposed epics and child plans.
  • Intent preservation after repeated local revisions.
  • Optional specialist reviewers: security, SRE, product, UX, copy, and business analysis.
  • Post-merge reflection with explicit discard of noisy learnings.

Works even better with metaswarm.

metaswarm owns the development lifecycle. metareview deepens the review gates inside that lifecycle.

Use metaswarm for issue intake, planning, orchestration, TDD-oriented work units, PR shepherding, and lifecycle coordination.

Use metareview at the decision points: artifact review before implementation, task-done review after local chunks, epic-ready review after child tasks, PR-ready review before branch handoff, and post-merge learning after the work lands.

When metaswarm, Superpowers, and Beads are already present, metareview extends those structures instead of creating parallel ones.

How it actually works

There is no server to trust and no proprietary review memory to rent back later. The harness is a Go CLI, plugin skills, Markdown rubrics, and repo-local artifacts.

CLI gates

metareview runs setup checks, context packing, lifecycle reviews, and post-merge learning from the repository checkout.

Markdown artifacts

Review logs and context packs are human-readable files under docs/metareview/.

Local state

Transient runs and findings use JSONL under .metareview/, keeping review mechanics inspectable.

Knowledge discovery

metareview looks for service inventories, Beads knowledge, prior sessions, GitHub context, and existing repo docs.

Plugin surfaces

Codex and Claude Code skills expose the same review gates to coding agents during normal work.

Git-native sync

Durable knowledge can travel with the repo through git instead of living only in a hosted reviewer.

Set it up

Install the CLI, or add the plugin to the agent runtime you already use.

npm CLI
npm install -g metareview
metareview setup --check
metareview status
Codex plugin
codex plugin marketplace add dsifry/metareview
codex plugin add metareview@metareview
Claude Code plugin
claude plugin marketplace add dsifry/metareview
claude plugin install metareview@metareview
source checkout
git clone https://github.com/dsifry/metareview.git
cd metareview
npm install
npm run build
./bin/metareview setup --check
